StoryFox is organized into shared cross-platform code, platform-specific UI, build scripts, and a Next.js landing site.

## Top-level layout

```
StoryFox/
├── StoryFoxApp.swift              # App entry point and navigation
├── Shared/                        # Cross-platform business logic
├── macOS/                         # macOS-specific UI and features
├── iOS/                           # iOS-specific UI (on hold)
├── Resources/                     # Assets, entitlements, Info.plist
├── scripts/                       # Build, release, and automation scripts
├── landing/                       # Next.js 15 landing page
├── release-notes/                 # HTML snippets for Sparkle appcast
├── project.yml                    # XcodeGen project spec (source of truth)
├── StoryFox.xcodeproj/            # Generated Xcode project (do not edit)
├── Makefile                       # Build automation targets
└── appcast.xml                    # Sparkle update feed
```

## Resources/

Assets, configuration, and app metadata:

```
Resources/
├── Assets.xcassets/
│   ├── AppIcon.appiconset/        # App icon
│   └── StoryFoxHero.imageset/     # About panel hero image
├── StoryFox.entitlements          # Sandbox entitlements (macOS)
├── StoryFox-iOS.entitlements      # iOS entitlements
├── StoryFox-Info.plist            # Sparkle public key
└── Diffusers/                     # (Deprecated: on-device diffusion models)
```

<Warning>
`xcodegen generate` can overwrite `StoryFox.entitlements` to an empty `<dict/>`. The `make dmg` target restores required entitlements. Always verify entitlements after regenerating the project.
</Warning>

## scripts/

Build and release automation:

| Script | Purpose |
|--------|----------|
| **generate.sh** | Regenerate Xcode project from `project.yml` |
| **build.sh** | Build Debug or Release configuration |
| **run.sh** | Build and launch app |
| **doctor.sh** | Verify toolchain and SDK readiness |
| **release.sh** | Full automated release pipeline |
| **inject-release-notes.sh** | Inject HTML into appcast.xml |

## project.yml

The **source of truth** for Xcode project configuration. Never edit `StoryFox.xcodeproj/project.pbxproj` directly.

### Key fields

```yaml project.yml
name: StoryFox

settings:
  base:
    SWIFT_VERSION: "6.0"
    MACOSX_DEPLOYMENT_TARGET: "26.0"

packages:
  MLXSwift:
    url: https://github.com/ml-explore/mlx-swift
    exactVersion: 0.30.6
  Sparkle:
    url: https://github.com/sparkle-project/Sparkle
    from: "2.8.1"

targets:
  StoryFox:
    type: application
    platform: macOS
    sources:
      - path: StoryFoxApp.swift
      - path: Shared
      - path: macOS
      - path: Resources/Assets.xcassets
    settings:
      base:
        MARKETING_VERSION: "1.9.2"
        CURRENT_PROJECT_VERSION: "33"
        PRODUCT_BUNDLE_IDENTIFIER: com.jakerains.StoryFox
    dependencies:
      - package: MLXSwift
      - package: Sparkle
```

### After editing project.yml

Always regenerate the Xcode project:

```bash
make generate
```

## Makefile targets

The `Makefile` provides convenient build commands:

| Target | Description |
|--------|-------------|
| `make help` | List all available targets |
| `make doctor` | Check toolchain and SDK readiness |
| `make build` | Build Debug for macOS |
| `make run` | Build and launch Debug app |
| `make build-release` | Build Release for macOS |
| `make run-release` | Build and launch Release app |
| `make dmg` | Full distribution pipeline: sign, notarize, staple, package DMG |
| `make clean` | Clean Xcode build artifacts |
| `make appcast` | Regenerate `appcast.xml` from DMGs in `dist/` |
| `make sparkle-setup` | One-time EdDSA key pair generation for Sparkle signing |
| `make purge-image-cache` | Remove cached Diffusers model data |

## File naming conventions

- **One primary type per file**, filename matches type name
  - `CreationViewModel.swift` defines `class CreationViewModel`
  - `BookFormat.swift` defines `enum BookFormat`
- **UpperCamelCase** for types and protocols
- **lowerCamelCase** for methods, properties, and variables
- **Platform suffixes** for platform-specific files:
  - `PDFRenderer+macOS.swift` — macOS-specific extension
  - `MacCreationView.swift` — macOS view
  - `IOSCreationView.swift` — iOS view (on hold)

## Code organization rules

1. **Keep `Shared/` compatible with both macOS and iOS** — use `#if os(macOS)` sparingly
2. **Platform-specific code lives in `macOS/` or `iOS/`** — never in `Shared/`
3. **One primary type per file** — exception: small helper structs/enums can live in the same file
4. **Use mark comments** for logical sections:
   ```swift
   // MARK: - Generation
   // MARK: - Actions
   // MARK: - Private Helpers
   ```
